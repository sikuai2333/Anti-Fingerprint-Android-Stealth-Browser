# 指纹浏览器专项整改指南（面向当前仓库）

> 文档目的：基于当前代码实现，给出“定位复核 + 问题清单 + 可落地整改方案”。

---

## 0. 项目定位复核（先把方向对齐）

### 0.1 当前项目实际定位

从代码看，这不是“通用浏览器内核项目”，而是一个 **WebView 容器型指纹浏览器**：

1. 以 `MainActivity + WebView` 为主入口，目标站点目前固定加载 `https://web.max.ru`。
2. 通过 JS 注入（`SpoofingEngine`）改写浏览器特征，降低 Android WebView 暴露度。
3. 通过代理层（SOCKS5/HTTP/VLESS）实现网络出口控制与连通性增强。
4. 通过权限拦截（摄像头/麦克风/定位）和 API 阻断降低信息泄露。

**结论**：项目定位应表述为“**特定业务场景下的抗检测 WebView 终端**”，而非“完整浏览器产品”。后续整改必须围绕“稳定性、安全性、可控性、可观测性”来做。

### 0.2 目标定义（整改后应达到）

- **可用**：目标站点可稳定登录和使用，兼容主流前端框架。
- **可控**：指纹模块可按域名/策略开关，不再“一刀切”。
- **可信**：网络与 TLS 安全基线达标，避免默认放行风险。
- **可维护**：核心逻辑可测试、可分层、可回归。

## 1. 审查结论总览（问题 -> 影响 -> 优先级）

| 编号 | 问题 | 影响 | 优先级 |
|---|---|---|---|
| P0-1 | `onReceivedSslError` 直接 `proceed()` | 中间人攻击风险，安全基线不合格 | P0 |
| P0-2 | VLESS 连接使用 trust-all `X509TrustManager` | TLS 信任链失效 | P0 |
| P0-3 | 全局允许 cleartext（Manifest + network config） | 明文流量降级面 | P0 |
| P1-1 | `MainActivity` 过重（UI+网络+策略混杂） | 维护成本高，改动易连锁破坏 | P1 |
| P1-2 | 线程模型分散（大量 `Thread{}`） | 生命周期泄漏、并发不可控 | P1 |
| P1-3 | Header/Tracker 模块有“定义未落地” | 设计与运行行为不一致 | P1 |
| P2-1 | 注入脚本体量巨大且耦合 | 回归难、站点兼容性不稳定 | P2 |
| P2-2 | 注入时机单一（started/finished） | 早期检测脚本覆盖不足 | P2 |
| P2-3 | 缺少自动化检测基线 | 变更后效果不可量化 | P2 |
| P3-1 | 代理能力有但策略层不足 | 多场景扩展慢 | P3 |
| P3-2 | 缺少统一可观测面板 | 运维与故障定位效率低 | P3 |

## 2. 详细整改方案（逐项给出可执行动作）

## 2.1 P0：安全基线整改（必须先做）

### P0-1 SSL 错误处理整改

**现状**：`PhantomWebViewClient.onReceivedSslError` 直接 `handler?.proceed()`。

**整改动作**：
1. 默认改为 `cancel()`。
2. 仅允许“开发模式 + 白名单域名”下手动放行。
3. 对用户显示清晰风险提示（域名、证书问题类型、是否继续）。

**验收标准**：
- 生产构建中，证书异常站点必须阻断。
- 开发构建可开关放行，且有明显日志标识。

---

### P0-2 VLESS TLS 信任链整改

**现状**：`ProxyManager.createVlessTunnel` 使用 trust-all 证书策略。

**整改动作**：
1. 删除自定义 trust-all `TrustManager`。
2. 使用系统默认信任管理器。
3. 可选新增证书 pinning（对固定上游节点）。
4. SNI 与 hostname 校验必须开启，失败即断开。

**验收标准**：
- 伪造证书环境下连接失败。
- 合法证书链可正常握手。

---

### P0-3 明文流量策略整改

**现状**：`usesCleartextTraffic=true`，`base-config cleartextTrafficPermitted=true`。

**整改动作**：
1. 全局改为 `false`。
2. 若业务必须明文，仅针对受控域名单独放行。
3. 记录明文请求告警日志，便于后续清理。

**验收标准**：
- 非白名单 HTTP 请求默认失败。
- 线上版本无意外明文链路。

---

## 2.2 P1：架构与可维护性整改

### P1-1 MainActivity 解耦

**现状**：`MainActivity` 同时处理 WebView、设置面板、代理连接、QR 识别、会话状态。

**整改动作**：
1. 拆分为 `WebContainerController`、`ProxySessionController`、`QrAssistantController`。
2. 引入 ViewModel 管理 UI 状态与事件。
3. `PhantomApp` 仅保留全局配置，不承载业务流程。

**验收标准**：
- `MainActivity` 代码行数显著下降（建议 < 600 行）。
- 核心业务可单测（不依赖 Activity 实例）。

---

### P1-2 并发模型统一

**现状**：多个模块中大量直接 `Thread {}`。

**整改动作**：
1. 使用 Coroutines（`viewModelScope` / `Dispatchers.IO`）。
2. 代理连接和 IP 刷新改为可取消任务。
3. 统一错误回传模型（sealed result）。

**验收标准**：
- 页面退出后无悬挂线程。
- 所有异步任务可追踪、可取消。

---

### P1-3 “定义未落地”模块清理

**现状**：`HeaderManager.applyHeaders`、`TrackerBlocker.shouldBlock` 有实现但主流程未接入。

**整改动作**：
1. 在 `shouldInterceptRequest` 中接入 Tracker 拦截与 Header 策略。
2. 若暂不接入，明确删除死代码，避免误导。
3. 建立“已启用能力清单”文档，防止宣传与实现错位。

**验收标准**：
- 代码中不存在长期未使用核心模块。
- 运行行为与文档声明一致。

---

## 2.3 P2：指纹策略工程化整改

### P2-1 脚本模块化

**现状**：`SpoofingEngine` 为超大拼接脚本，改动风险高。

**整改动作**：
1. 按能力拆分：UA/CH、Canvas、WebGL、Audio、RTC、Permissions、Timezone。
2. 每个模块独立版本号与开关。
3. 使用模板化脚本生成器，避免手写大字符串。

**验收标准**：
- 任一模块可单独关闭而不影响其他模块。
- 模块级回归报告可输出。

---

### P2-2 注入时机增强

**现状**：主要在 `onPageStarted/onPageFinished` 注入。

**整改动作**：
1. 优先启用 DocumentStart 级注入（可用时）。
2. 对 SPA 路由切换追加轻量补注入。
3. 注入失败要有统计与重试策略（限次）。

**验收标准**：
- 早期探针脚本命中率提升。
- 首屏脚本执行顺序可观测。

---

### P2-3 兼容性回归体系

**现状**：缺少自动化效果验证。

**整改动作**：
1. 建立目标站点 + 检测站点回归清单。
2. 固化指标：加载成功率、登录成功率、脚本异常率、平均首屏耗时。
3. 每次改动执行 A/B 比较并留档。

**验收标准**：
- 每次发布有量化回归报告。
- 兼容性下降可快速定位到具体模块。

---

## 2.4 P3：扩展与产品化整改

### P3-1 策略引擎化

**整改动作**：
1. 支持按域名加载不同 spoof/profile。
2. 支持代理分流规则（直连/节点A/节点B）。
3. 支持“低兼容模式/高隐蔽模式”预设。

### P3-2 可观测平台

**整改动作**：
1. 增加本地调试面板：当前 UA、CH、WebGL、代理出口、权限拦截计数。
2. 错误日志结构化（注入失败、连接失败、证书失败分类）。
3. 提供导出诊断包（脱敏）。

---

## 3. 建议实施顺序（8~12 周）

### 第 1 阶段（第 1~2 周）：安全止血
- 完成 P0-1 / P0-2 / P0-3。
- 发布仅包含安全修复的补丁版本。

### 第 2 阶段（第 3~6 周）：架构整顿
- 完成 P1-1 / P1-2 / P1-3。
- 建立统一异步与错误处理规范。

### 第 3 阶段（第 7~10 周）：策略工程化
- 完成 P2-1 / P2-2 / P2-3。
- 上线可回归、可度量的指纹策略流程。

### 第 4 阶段（第 11~12 周）：扩展能力
- 推进 P3-1 / P3-2。
- 为多场景接入做准备。

---

## 4. 风险与边界说明（必须明确）

1. **技术边界**：WebView 方案本质上仍受 Android 平台特征约束，不可能 100% 等价桌面 Chromium。
2. **业务边界**：目标是“降低识别概率 + 提高可用性”，不是“绝对不可检测”。
3. **合规边界**：反检测技术应仅用于合法授权场景，需符合应用商店政策和当地法律法规。

## 6. 最终判定标准（整改是否成功）

当同时满足以下条件，可视为整改完成：

1. 生产环境不再存在默认 TLS 放行和全局明文策略。
2. 主流程结构清晰，核心模块可测试可追踪。
3. 指纹策略具备模块开关与量化回归能力。
4. 文档声明与实际运行能力一致，无“宣传功能未接入”情况。
