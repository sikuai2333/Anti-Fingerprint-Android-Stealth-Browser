# 变更影响与回归建议

## 一、变更影响评估

### 1. 安全层面（正向）
- 默认拒绝 SSL 异常，降低中间人攻击风险。
- 禁用 cleartext 与 mixed content，缩小降级攻击面。
- VLESS TLS 从 trust-all 改为系统信任链 + 主机名校验，链路可信度提升。

### 2. 稳定性层面（双向）
- `ProxyManager` 协程化后，任务取消更可控，后台残留线程风险下降。
- `SpoofingEngine` 模块化提高了可维护性，但模块组合变化可能影响少量站点兼容性，需要回归验证。

### 3. 兼容性层面（需重点关注）
- `SAFE_NO_SPOOF` 等域名策略可能导致目标站点行为变化（例如某些站点检测逻辑回退到移动特征）。
- Tracker 拦截可能影响部分站点统计脚本与登录流程中的第三方依赖。

### 4. 可观测性层面（正向）
- `SpoofMetrics` + 状态页 profile/route 展示，可更快定位“注入失败/路由异常”的根因。

## 二、建议重点回归场景

### A. 登录与核心流程
1. `https://web.max.ru` 首屏加载与登录。
2. 代理连接后登录（SOCKS5/HTTP/VLESS）。
3. 代理断开后继续访问（确认状态提示正确）。

### B. 安全策略验证
1. 访问证书异常站点，确认拦截而非放行。
2. 抓包确认不再出现 cleartext 主请求。
3. 校验 VLESS 握手在错误证书下失败。

### C. 指纹与注入验证
1. 查看设置页 `SpoofMetrics`：Attempts 增长、SuccessRate 稳定。
2. 验证不同域名 profile 命中是否符合预期（STRICT/BALANCED/SAFE）。
3. 检查注入后页面 JS 错误率是否上升。

### D. 生命周期与资源
1. 前后台切换时 WebView 网络阻断/恢复是否正常。
2. `destroyOnMinimize` 开启时是否可自动重建。
3. 反复打开设置页与切换标签，观察 UI 状态是否一致。

---

## 三、回归执行建议

- 每次发布前至少执行一轮：
  - 构建检查：`assembleDebug`
  - 任务检查：`testDebugUnitTest`（当前为 `NO-SOURCE`）
  - 手工检查：本文件“二、重点回归场景”A/B/C 各至少 1 条
- 若出现兼容性回退：
  1. 先看 profile 命中与 route 展示；
  2. 再看 `SpoofMetrics` 是否异常；
  3. 最后回滚单个 spoof 模块验证影响范围。
